Design Philosophy

NewTon DC Tournament Manager is built around a single overriding principle: a tournament that has started must never lose data or end up in an invalid state. Every architectural decision follows from that.

This means no dynamic calculations that could produce different results depending on execution order, no mutable shared state that different code paths can corrupt, and no operations that can half-complete. Instead, NewTon uses hardcoded lookup tables as the single source of truth for all bracket logic, and a transaction-based history system that logs every match action and enables surgical undo.

The result is a system that can survive almost any failure — a misclick, a browser crash, a power cut mid-tournament — and be restored to an exact, consistent state.

Single Source of Truth

The most critical architectural decision in NewTon is how bracket progression works. Many systems calculate where a player should go next at runtime — using formulas, round arithmetic, or conditional logic. NewTon does not.

Instead, all bracket progression is defined in a single constant object called MATCH_PROGRESSION, hardcoded in clean-match-progression.js:

const MATCH_PROGRESSION = {
    8:  { /* all rules for 8-slot brackets  */ },
    16: { /* all rules for 16-slot brackets */ },
    32: { /* all rules for 32-slot brackets */ }
};

Every match — on the Frontside and Backside — has an explicit entry that defines exactly where the winner goes and where the loser goes:

'FS-1-1': {
    winner: ['FS-2-1', 'player1'],
    loser:  ['BS-1-1', 'player1']
}

There is only one function that moves players between matches — advancePlayer() — and it does nothing but look up the rule and apply it. No calculations. No special cases. No branches.

Why this matters

  • Immutable — the rules are hardcoded and never changed at runtime. They cannot be corrupted by user actions or application state.
  • Complete — every possible match outcome in every bracket size is explicitly defined. There are no gaps, no fallbacks, no guesses.
  • Deterministic — given the same match result, the same progression happens every single time, with no variation.
  • Auditable — because there is one code path, every advancement is traceable. You always know exactly how a player got to where they are.
  • Protected — any operation that can directly modify the bracket structure or transaction log is deliberately hidden behind the Developer Console, keeping destructive tools out of reach during normal tournament operation.

Transaction History & Undo

Every match action in NewTon is recorded as a transaction in a per-tournament log. The tournament's current state is not just a snapshot — it is the sum of every recorded action that has been applied to it.

This approach — known as event sourcing — has two important consequences:

  • Undo is surgical — because every action is discrete and logged, individual match results can be reversed without affecting the rest of the tournament. NewTon does not need to restore a full snapshot; it replays the history minus the action being undone.
  • Auto-recovery — if the application state is ever lost or corrupted, the entire bracket can be rebuilt by replaying the transaction log from the beginning. The log is the ground truth.

The transaction log is capped at 1,000 entries, visible and manageable via the Developer Console. The Console also shows a Transaction Health indicator so operators can see at a glance how much of the limit is in use.

Data Separation

NewTon maintains strict isolation between three categories of data, each stored independently and never mixed:

CategoryWhat it containsStorage key
Tournament dataPlayers, matches, bracket, placements, statustournament_{id}
Transaction historyFull audit log of every match actiontournament_{id}_history
Global configurationPoints, legs, lanes, UI preferencesdartsGlobalConfig

Configuration is never saved with a tournament. Tournament exports contain only the tournament's own data — not the config of the machine it was exported from. This means a tournament exported from one club's setup can be imported at another without any configuration bleed.

When a tournament is deleted, all of its keys are removed completely. No residue, no contamination between tournaments.

Offline-First Persistence

NewTon has no server dependency for any of its core functionality. All data lives in the browser's localStorage — a sandboxed, domain-isolated storage area that persists across browser restarts and reboots without any network connection.

Every state change is saved immediately. There is no "save button" because there is no moment where unsaved state exists. If the browser crashes mid-tournament, the last saved state is fully intact and consistent.

  • No internet required after the initial download
  • No account, no login, no sync service
  • Export/import via JSON for backups, sharing, and moving between devices
  • Optional REST API for self-hosters who want shared tournament access — disabled by default, never automatic

See the Privacy Architecture page for the full storage model.

Strengths

  • Correctness — hardcoded lookup tables eliminate an entire class of calculation bugs. The rules cannot be wrong at runtime because they are not computed at runtime.
  • Crash resistance — immediate persistence, transaction history, and auto-recovery from replay mean the application survives almost any failure scenario.
  • Maintainability — clear separation between progression rules, tournament data, and configuration means changes in one area cannot corrupt another.
  • Transparency — self-documenting match IDs (FS-2-1 = Frontside Round 2 Match 1, BS-FINAL = Backside Final), an auditable transaction log, and open-source code mean the system's behaviour is always inspectable.
  • Extensibility — adding a new bracket size requires only a new lookup table entry. No core logic changes needed.

Known Limitations

Honesty is part of reliability. These are the known constraints of the current architecture:

  • Bracket templates are fixed — NewTon supports any number of participants from 4 to 32. Unused slots are automatically filled with byes that advance without playing, so a 13-player tournament runs cleanly in a 16-slot bracket with 3 invisible byes. The constraint is the upper limit of 32 participants, and the three available template sizes (8, 16, 32 slots). Adding a larger template requires a new hardcoded lookup table.
  • localStorage limits — browsers typically allow 5–10 MB per domain. A 32-player tournament with full match history uses around 110 KB. Very large numbers of tournaments with extensive undo history could approach limits; the Developer Console monitors this in real time.
  • Single-user design — NewTon is designed for one operator per tournament. There is no multi-device sync or conflict resolution for simultaneous edits. The optional REST API enables sharing, but not live collaboration.
